# Claude Code 多角色协作流水线 v5 设计方案

## 概述

本文档记录 v5 版本的设计决策，基于对 v4 的逻辑漏洞分析提出。v5 修复了 6 个残余逻辑漏洞，新增 3 个 AutoStep，所有新增语义检测均使用开源工具实现（无新增 LLM 调用），总执行单元从 35 增加到 38。

---

## v5 核心改进要点

- 修复漏洞 K：新增 Phase 2.7 Contract Semantic Validator（AutoStep），使用 Spectral + 自定义脚本封堵"格式合法但语义错误"的 OpenAPI Schema，防止 Contract Formalizer 的类型/字段错误静默传递到 Phase 3
- 修复漏洞 L：新增 Phase 4a.1 Test Failure Mapper（AutoStep），分析测试失败的 Builder 责任归属，实现精确回退而非全体回退
- 修复漏洞 M：明确正常流程下 `api_changed: false` 时 Phase 5 的执行策略（新增 `phase_5_mode: changelog_only`），消除状态机空白路径
- 修复漏洞 N：修正第 8 节目录结构中 `.pipeline/artifacts/` 重复定义，统一 adr-draft.md 路径
- 修复漏洞 O：gate-b-review.json 新增 `assumption_dispositions` 字段，将 Gate B 对假设的处置决策从自然语言 comments 升级为结构化记录，支持机械流转
- 修复漏洞 P：perf-report.json 新增 `sla_violated` 字段，Optimizer 明确发现 SLA 违规时直接触发 Phase 3 回退，不等待 Gate D
- 新增 Phase 0.5 Requirement Completeness Checker（AutoStep）：需求文档进入 Gate A 前的格式与完整性机械门禁
- 新增 Phase 2.7 Contract Semantic Validator（AutoStep）：封堵 Phase 2.6 无法检测的语义错误，全部使用开源工具（Spectral + 脚本），无 LLM
- 新增 Phase 4a.1 Test Failure Mapper（AutoStep）：测试失败后精确映射责任 Builder，降低多 Builder 场景下的无辜回退成本

---

## 漏洞修复详细设计

### 漏洞 K — Phase 2.7 Contract Semantic Validator（AutoStep）

**问题根因**：Phase 2.6 Schema Completeness Validator 只验证 OpenAPI 格式合法性和数量，不验证语义正确性。Contract Formalizer（LLM）可能生成"格式合法但语义错误"的 Schema，例如将整数类型定义为 string、将路径参数遗漏 required 标记、将创建操作使用 GET 方法等。此类错误在 Phase 3.7 Contract Compliance Checker 才会被发现，届时整个 Phase 3 实现已完成，回退成本极高。

**修复方案**：新增 Phase 2.7 Contract Semantic Validator AutoStep，在 Phase 2.6 PASS 后、Phase 3 前执行，使用开源工具进行语义规则校验。

**工具栈（全部开源）**：
- `@stoplight/spectral-core` + `@stoplight/spectral-openapi`：RESTful 语义规则
- `@stoplight/spectral-owasp-rules`：API 安全基础规则
- 自定义比对脚本：`tasks.json contracts[].definition` 字段 ↔ OpenAPI Schema properties 的类型/名称比对

**执行内容**：

1. **Spectral 规则校验**（使用 `@stoplight/spectral-openapi` + `@stoplight/spectral-owasp-rules`）：
   - 路径参数必须标记为 `required: true`（`oas3-path-params`）
   - 每个 operation 必须有 `operationId`（`operation-operationId`）
   - 每个 operation 必须有 2xx 响应定义（`operation-success-response`）
   - GET / HEAD / DELETE 不得有 requestBody（`oas3-no-request-body-for-get`）
   - API Key 不得在 query 参数中传递（OWASP 规则）

2. **tasks.json ↔ OpenAPI 字段比对脚本**：
   - 提取 `tasks.json contracts[].definition.response` 中每个字段的名称和类型声明
   - 与对应 `contracts/contract-N.openapi.json` 的 `properties` 逐字段比对
   - 检测字段名不匹配或类型不一致（如 `integer` vs `string`）

**产物** `.pipeline/artifacts/contract-semantic-report.json`：

```json
{
  "autostep": "ContractSemanticValidator",
  "timestamp": "2025-01-01T00:00:45Z",
  "spectral_violations": [
    {
      "contract_id": "contract-1",
      "file": "contracts/contract-1.openapi.json",
      "rule": "oas3-path-params",
      "message": "路径参数 'id' 未标记为 required",
      "severity": "ERROR",
      "line": 12
    }
  ],
  "field_type_mismatches": [
    {
      "contract_id": "contract-1",
      "field": "id",
      "tasks_json_type": "integer",
      "openapi_type": "string",
      "severity": "ERROR"
    }
  ],
  "warnings": [],
  "overall": "FAIL"
}
```

**流转规则**：
- 任意 `severity: ERROR` → 回退 Phase 2.5（Contract Formalizer 基于报告修正 Schema）
- 仅有 `severity: WARN` → 不阻断，追加到 Gate C Inspector 的参考输入上下文中
- 无任何问题 → 进入 Phase 3（并行实现）

### 漏洞 L — Phase 4a.1 Test Failure Mapper（AutoStep）+ test-report.json 增强

**问题根因**：Phase 4a 测试失败后，test-report.json 只记录测试名和错误信息，未关联 Builder 责任。Pilot 无法判断应回退哪些 Builder，当前文档的"回退含 3.1→3.6→Gate C→3.7"语义不清晰——实际上只能整体回退所有 Builder，多个无辜 Builder 被迫重新实现，浪费成本。

**修复方案**：新增 Phase 4a.1 Test Failure Mapper AutoStep，在 Phase 4a FAIL 后、Phase 3 回退前执行，分析测试失败的文件调用链，映射到责任 Builder。

**执行内容**：
1. 解析 `test-report.json` 中 failures 的测试名
2. 从覆盖率工具的 `coverage.lcov` 输出（或 Istanbul/nyc 的 JSON 覆盖率报告）中提取每个失败测试涉及的源文件路径
3. 与 `impl-manifest.json` 中各 Builder 的 `files_changed` 列表交叉比对
4. 推断每个失败测试的 `responsible_builders`
5. 合并所有失败测试的责任 Builder 集合，输出 `builders_to_rollback`

**置信度规则**：
- `HIGH`：失败测试涉及的所有文件均可唯一归属单一 Builder
- `LOW`：涉及多个 Builder 共享的文件（如共享 utils），无法唯一归属
- `UNKNOWN`：无法从覆盖率数据中提取涉及文件（降级到全体回退）

**产物** `.pipeline/artifacts/failure-builder-map.json`：

```json
{
  "autostep": "TestFailureMapper",
  "timestamp": "2025-01-01T00:07:30Z",
  "failure_mappings": [
    {
      "test_name": "test_get_user_not_found",
      "involved_files": ["src/routes/user.ts", "src/middleware/auth.ts"],
      "responsible_builders": ["Builder-Backend", "Builder-Security"],
      "confidence": "HIGH"
    },
    {
      "test_name": "test_create_order_validation",
      "involved_files": ["src/services/order.ts"],
      "responsible_builders": ["Builder-Backend"],
      "confidence": "HIGH"
    }
  ],
  "builders_to_rollback": ["Builder-Backend", "Builder-Security"],
  "unmapped_failures": [],
  "overall": "MAPPED"
}
```

**流转规则**：
- `unmapped_failures` 为空（`overall: MAPPED`）→ 只回退 `builders_to_rollback` 中的 Builder，其余 Builder 结果保留
- `unmapped_failures` 不为空（`overall: PARTIAL_MAPPED`）→ 降级：回退所有 Builder（与 v4 行为一致），并在日志中记录 `[WARN] 测试失败部分无法映射到 Builder，执行全体回退`

### 漏洞 M — Phase 5 策略完整定义（`phase_5_mode`）

**问题根因**：状态机中只定义了 `api_changed: true` 和 `api_changed: false + hotfix` 两个分支，正常流程下 `api_changed: false` 的路径缺失。实际上，即使 API 没有变更，代码实现变更也应体现在 CHANGELOG 中。

**修复方案**：在 `state.json` 新增 `phase_5_mode` 字段，由 API Change Detector AutoStep 写入，Documenter 和 Phase 5.1 Changelog Consistency Checker 根据此字段决定执行内容。

**完整策略矩阵**：

| 场景 | `api_changed` | `mode` | `phase_5_mode` | Phase 5 执行内容 | Phase 5.1 是否运行 |
|------|--------------|--------|---------------|----------------|--------------------|
| 正常流程，API 有变更 | true | normal | `full` | API 文档 + CHANGELOG + ADR 最终化 | 是 |
| 正常流程，API 无变更 | false | normal | `changelog_only` | **仅更新 CHANGELOG**，跳过 API 文档 | 是（验证 CHANGELOG 覆盖 impl-manifest） |
| Hotfix，API 有变更 | true | hotfix | `full` | API 文档 + CHANGELOG + ADR 最终化 | 是 |
| Hotfix，API 无变更 | false | hotfix | `skip` | 跳过整个 Phase 5 | 否 |

**state.json 新增字段**：

```json
{
  "phase_5_mode": "changelog_only"
}
```

**api-change-report.json 新增字段**：

```json
{
  "autostep": "APIChangeDetector",
  "api_changed": false,
  "phase_5_mode": "changelog_only",
  "documentation_required": false,
  "changelog_required": true
}
```

**Phase 5.1 在 `changelog_only` 模式下的调整**：跳过 API 路径关键词检查（因无 API 变更），只验证 CHANGELOG 包含 `impl-manifest.json` 中 `files_changed` 涉及模块的变更描述。

### 漏洞 O — gate-b-review.json 新增 `assumption_dispositions`

**问题根因**：Phase 2.1 Assumption Propagation Validator 将未覆盖假设附加给 Auditor-Biz，但 gate-b-review.json 模板中没有字段记录 Auditor-Biz 对每条假设的处置决策。处置结果只存在于自然语言 comments 中，Pilot 无法机械判断假设是否被正式接受，下游 Builder 也无法引用。

**修复方案**：gate-b-review.json 新增 `assumption_dispositions` 数组，Auditor-Biz 对 `assumption-propagation-report.json` 中每条 `uncovered` 假设必须明确标注处置。

**gate-b-review.json 新增字段**：

```json
{
  "gate": "B",
  "assumption_dispositions": [
    {
      "assumption": "第三方支付回调格式遵循标准 Webhook 格式",
      "source": "requirement.md:行42",
      "disposition": "ACCEPTED",
      "auditor": "Auditor-Biz",
      "note": "与支付供应商确认后可接受此假设，无需 Planner 补充任务"
    },
    {
      "assumption": "用户量不超过 10 万",
      "source": "requirement.md:行18",
      "disposition": "REQUIRE_PLANNER_COVERAGE",
      "auditor": "Auditor-Tech",
      "note": "需要 Planner 在 tasks.json 中增加限流和分页任务以覆盖此假设"
    }
  ],
  "results": [...],
  "overall": "FAIL"
}
```

**disposition 枚举值**：
- `ACCEPTED`：接受假设，无需 Planner 补充任务，后续 Builder 可直接基于此假设实现
- `REQUIRE_PLANNER_COVERAGE`：要求 Planner 在 tasks.json 中增加对应任务或 notes 引用

**流转规则**：
- 任意假设的 `disposition: REQUIRE_PLANNER_COVERAGE` → Gate B FAIL，`rollback_to: phase-2`（Planner 补充任务）
- 所有假设均 `ACCEPTED` 且其他审查项 PASS → Gate B PASS（假设处置不影响其他审查结论）
- 若 `assumption-propagation-report.json` 中 `uncovered` 为空，`assumption_dispositions` 数组为空（无需处置）

### 漏洞 P — Optimizer 直接触发 Phase 3 回退

**问题根因**：Optimizer 是定量产出 perf-report.json 的角色，其 SLA 判断是纯机械比对（实测 p99 vs config.json SLA 阈值），不需要 Gate D 的 Auditor-QA 主观审批即可确定回退结论。当前设计中 Optimizer 无直接回退机制，明确的 SLA 违规需要等待 Gate D，增加了不必要的延迟。

**修复方案**：perf-report.json 新增 `sla_violated` 和 `rollback_to` 字段。Pilot 检测到 `sla_violated: true` 时，直接回退 Phase 3，不进入 Gate D。

**perf-report.json 新增字段（SLA 违规场景）**：

```json
{
  "sla": { "p99_latency_ms": 200 },
  "results": {
    "api_get_user": { "p50_ms": 120, "p99_ms": 850, "verdict": "FAIL" }
  },
  "slow_queries": [
    { "query": "SELECT * FROM users WHERE email = ?", "avg_ms": 620, "file": "src/services/user.ts:84" }
  ],
  "sla_violated": true,
  "rollback_reason": "api_get_user p99=850ms 超出 SLA 上限 200ms，疑似 N+1 查询，见 slow_queries",
  "rollback_to": "phase-3",
  "overall": "FAIL"
}
```

**perf-report.json（SLA 达标场景，与 v4 兼容）**：

```json
{
  "sla": { "p99_latency_ms": 200 },
  "results": {
    "api_get_user": { "p50_ms": 45, "p99_ms": 185, "verdict": "PASS" }
  },
  "slow_queries": [],
  "sla_violated": false,
  "overall": "PASS"
}
```

**Pilot 行为**：
- `sla_violated: true` → 直接回退 Phase 3，将 `slow_queries` 和 `rollback_reason` 注入对应 Builder 的输入上下文；递增对应 Builder 的 `builder_attempt_counts`
- `sla_violated: false` → 正常进入 Gate D，由 Auditor-QA 做最终验收（含主观判断）

## 新 AutoStep 详细设计

### Phase 0.5 Requirement Completeness Checker（AutoStep）

**类型**：AutoStep

**位置**：Phase 0（Clarifier）完成后，Gate A 审核前

**价值**：让 Gate A 的 Auditor 聚焦内容审查而非格式问题；确保 Phase 2.1 Assumption Propagation Validator 的关键词提取输入源格式可靠。

**执行内容**：

1. **必填 Section 检查**：验证 requirement.md 包含以下 Section 标题且内容非空：
   - `## 功能描述`
   - `## 用户故事`
   - `## 验收标准`
   - `## 范围边界`

2. **关键项检查**：确认 `[CRITICAL-UNRESOLVED]` 数量为 0（Pilot 在 Phase 0 结束时已判断，此处为二次机械确认）

3. **假设格式检查**：所有 `[ASSUMED:...]` 条目必须符合正则 `\[ASSUMED:[^\]]+\]`，确保 Phase 2.1 的关键词提取不受格式异常干扰

4. **最小字数检查**：需求文档总字数 ≥ `config.json` 中 `requirement_min_words`（默认 200）

**产物** `.pipeline/artifacts/requirement-completeness-report.json`：

```json
{
  "autostep": "RequirementCompletenessChecker",
  "timestamp": "2025-01-01T00:00:05Z",
  "sections_check": {
    "功能描述": "PRESENT",
    "用户故事": "PRESENT",
    "验收标准": "PRESENT",
    "范围边界": "MISSING"
  },
  "critical_unresolved_count": 0,
  "assumed_items_count": 2,
  "assumed_items_valid_format": true,
  "word_count": 450,
  "word_count_threshold": 200,
  "overall": "FAIL"
}
```

**流转规则**：
- `overall: FAIL` → 回退 Phase 0（Clarifier 补充缺失内容），不进入 Gate A
- `overall: PASS` → 进入 Gate A

**config.json 新增配置**：

```json
{
  "requirement_completeness": {
    "required_sections": ["功能描述", "用户故事", "验收标准", "范围边界"],
    "min_words": 200,
    "abort_on_critical_unresolved": true
  }
}
```

**修复范围：** 漏洞 Q 至 漏洞 W，共 7 处，均为精准 Edit 操作，按文档章节顺序从前到后执行。

## 漏洞 Q：Resolver conditions 字段无机械执行路径

### 问题

Gate A 的 `resolver_verdict` 示例中存在：

```json
{
  "verdict": "PASS",
  "rollback_to": null,
  "conditions": "Architect 需在 Proposal 中将 WebSocket 改为 SSE 方案"
}
```

Pilot 收到 PASS 后直接进入下一阶段，**conditions 是纯文本字符串，没有任何机制验证其是否被执行**。Resolver 的条件承诺成了空话。这与设计原则"产物驱动流转"相矛盾——条件的执行取决于下游 Agent 是否读到了这段文字，而非机械验证。

### 修复方案

**1. resolver_verdict 新增 `conditions_checklist` 结构化字段**

将 `conditions` 字符串升级为可机械流转的数组：

```json
{
  "reviewer": "Resolver",
  "verdict": "PASS",
  "rollback_to": null,
  "conditions": "Architect 需在 Proposal 中将 WebSocket 改为 SSE 方案",
  "conditions_checklist": [
    {
      "target_agent": "Architect",
      "target_phase": "phase-1",
      "requirement": "将 Proposal 中的 WebSocket 替换为 SSE 方案，并更新影响面分析",
      "verification_method": "grep",
      "verification_pattern": "SSE|Server-Sent Events",
      "verification_file": ".pipeline/artifacts/proposal.md"
    }
  ]
}
```

`verification_method` 枚举：
- `grep`：在 `verification_file` 中搜索 `verification_pattern` 是否出现
- `exists`：检查 `verification_file` 是否存在且非空
- `field_value`：检查某 JSON 文件中特定字段的值（如 `{"file": "...", "field": "$.tech_stack", "expected": "SSE"}`）

**2. Pilot 在放行前执行条件验证**

当 `resolver_verdict.conditions_checklist` 非空时，Pilot 在将状态推进到下一阶段前：

1. 通知目标 Agent 重新处理（携带 conditions_checklist 作为输入约束）。
2. 目标 Agent 完成后，Pilot 逐条执行 `verification_method` 机械检查。
3. 所有条目通过 → 写入 `resolver-conditions-check.json`（`overall: PASS`），推进到下一阶段。
4. 任意条目失败 → `overall: FAIL`，Pilot 记录 `[WARN] Resolver 条件未满足，触发 rollback_to: <target_phase>`，回退到 `target_phase`。

**3. conditions_checklist 为空时的行为**

若 `conditions_checklist` 为空数组（Resolver 无附加条件）则跳过验证，直接推进。兼容存量的纯文本 `conditions` 字段（保留供人阅读，不参与机械流转）。

**产物：** `.pipeline/artifacts/resolver-conditions-check.json`（仅在 conditions_checklist 非空时生成）

```json
{
  "gate": "A",
  "resolver_conditions_check": true,
  "timestamp": "2025-01-01T00:00:08Z",
  "checks": [
    {
      "target_agent": "Architect",
      "requirement": "将 Proposal 中的 WebSocket 替换为 SSE 方案",
      "verification_method": "grep",
      "verification_pattern": "SSE|Server-Sent Events",
      "verification_file": ".pipeline/artifacts/proposal.md",
      "result": "PASS",
      "matched_lines": 3
    }
  ],
  "overall": "PASS"
}
```

**影响范围：**
- Section 2.4（Resolver 机制说明）：新增 conditions_checklist 说明
- Gate A 章节（Section 5）：更新 resolver_verdict 示例
- Section 7.1 状态机：在 RESOLVER 节点后新增 RESOLVER_CONDITIONS_CHECK 状态

## 漏洞 R：Phase 4a.1 confidence 字段未参与流转决策

### 问题

Phase 4a.1 对每条 `failure_mapping` 设置 `confidence: HIGH/LOW/UNKNOWN`，但流转规则只检查 `unmapped_failures` 是否为空：

- `unmapped_failures` 为空 → 精确回退 `builders_to_rollback`
- `unmapped_failures` 不为空 → 全体回退

`confidence` 字段完全不影响流转。一个 `confidence: LOW`（归属不确定）的映射和 `confidence: HIGH` 的映射被同等对待，导致不确定的 Builder 被错误回退。

### 修复方案

**flow 规则细化：**

```
failure_mappings 中所有映射均为 HIGH confidence 且 unmapped_failures 为空
  → overall: PRECISE_MAPPED
  → 只回退 builders_to_rollback（精确回退，不含 LOW 归属的 Builder）

failure_mappings 中存在 LOW confidence 的映射 且 unmapped_failures 为空
  → overall: LOW_CONFIDENCE_MAPPED
  → LOW confidence 映射的 responsible_builders 升级为 full rollback（整合到全体回退）
  → 等同 PARTIAL_MAPPED：回退所有 Builder
  → 日志记录 [WARN] 存在 LOW 置信度映射，执行保守全体回退

unmapped_failures 不为空（无论 confidence 如何）
  → overall: PARTIAL_MAPPED
  → 回退所有 Builder（与原行为一致）
```

**产物 JSON 更新（新增 `builders_high_confidence` 字段）：**

```json
{
  "autostep": "TestFailureMapper",
  "failure_mappings": [
    {
      "test_name": "test_get_user_not_found",
      "involved_files": ["src/routes/user.ts"],
      "responsible_builders": ["Builder-Backend"],
      "confidence": "HIGH"
    },
    {
      "test_name": "test_auth_middleware",
      "involved_files": ["src/middleware/auth.ts", "src/config/security.ts"],
      "responsible_builders": ["Builder-Security", "Builder-Backend"],
      "confidence": "LOW"
    }
  ],
  "builders_to_rollback": ["Builder-Backend", "Builder-Security"],
  "builders_high_confidence": ["Builder-Backend"],
  "unmapped_failures": [],
  "overall": "LOW_CONFIDENCE_MAPPED"
}
```

**Pilot 行为说明（新增）：**

- `overall: PRECISE_MAPPED` → 只回退 `builders_high_confidence`，其余 Builder 的实现结果保留。
- `overall: LOW_CONFIDENCE_MAPPED` → 降级为全体回退，日志记录原因。
- `overall: PARTIAL_MAPPED` → 全体回退（行为不变）。

**影响范围：**
- Phase 4a.1 章节（Section 5）：更新流转规则和产物 JSON 示例
- Section 7.1 状态机：更新 PHASE_4A_1_TEST_FAILURE_MAPPER 分支

## 漏洞 S：Phase 0.5 Section 标题层级与 requirement.md 格式不一致

### 问题（高危）

Phase 0.5 的执行内容第 1 条：

> 验证 requirement.md 包含以下 Section 标题且内容非空：
> - `## 功能描述`
> - `## 用户故事`
> - `## 验收标准`
> - `## 范围边界`

但 Phase 0 定义的 requirement.md 格式为：

```markdown
## 最终需求定义         ← H2（顶级节）

### 功能描述            ← H3（子节）
### 用户故事            ← H3
### 业务规则            ← H3
### 范围边界（包含 / 不包含）  ← H3
### 验收标准            ← H3
### 非功能需求（业务侧）  ← H3
```

Phase 0.5 搜索 `## 功能描述`（H2），但文档中只有 `### 功能描述`（H3）。**对所有合法的 requirement.md 都会输出 FAIL**，Phase 0.5 永远无法通过。

此外：
- Phase 0.5 检查 `## 用户故事` 但格式中为 `### 用户故事`（H3）
- "范围边界" 在格式中为 "范围边界（包含 / 不包含）"，精确匹配会失败
- Phase 0.5 未检查 `### 业务规则`（requirement.md 格式有此节）

### 修复方案

**修改 Phase 0.5 Section 1 的检查逻辑：**

将 H2 检查替换为"在 `## 最终需求定义` 区块内检查 H3 子节"：

1. 首先定位 `## 最终需求定义`（H2 标题），提取该节下的所有内容（到下一个 H2 或文件末尾）。
2. 在提取内容中查找以下 H3 标题（使用前缀匹配，允许标题后有括号补充说明）：
   - `### 功能描述`
   - `### 用户故事`
   - `### 业务规则`
   - `### 范围边界`（匹配"范围边界"前缀即可，不要求完整匹配）
   - `### 验收标准`
3. 以上 5 个 H3 全部存在且非空 → `sections_check.overall: PASS`。
4. 任意缺失 → `sections_check.overall: FAIL`，列出缺失项。

**更新 Phase 0.5 产物 JSON 示例（sections_check 字段细化）：**

```json
{
  "autostep": "RequirementCompletenessChecker",
  "timestamp": "2025-01-01T00:00:05Z",
  "sections_check": {
    "最终需求定义_section_found": true,
    "功能描述": "PRESENT",
    "用户故事": "PRESENT",
    "业务规则": "PRESENT",
    "范围边界": "PRESENT",
    "验收标准": "MISSING"
  },
  "critical_unresolved_count": 0,
  "assumed_items_count": 2,
  "assumed_items_valid_format": true,
  "word_count": 450,
  "word_count_threshold": 200,
  "overall": "FAIL"
}
```

**同步更新 config.json 中的 `requirement_completeness.required_sections`：**

```json
"requirement_completeness": {
  "parent_section": "## 最终需求定义",
  "required_sections": ["### 功能描述", "### 用户故事", "### 业务规则", "### 范围边界", "### 验收标准"],
  "section_match_mode": "prefix",
  "min_words": 200,
  "abort_on_critical_unresolved": true
}
```

**影响范围：**
- Phase 0.5 章节（Section 5）：更新执行内容第 1 条 + 产物 JSON 示例
- Section 9 config.json：更新 `requirement_completeness` 配置块

## 漏洞 T：Gate D JSON schema 缺少结构化 rollback_to 字段

### 问题

Gate A（gate-a-review.json）和 Gate C（gate-c-review.json）均在 JSON schema 中包含 `rollback_to` 字段，供 Pilot 机械解析回退目标。

但 Gate D 的 JSON 示例：

```json
{
  "gate": "D",
  "test_verdict": "PASS",
  "coverage_verdict": "PASS",
  "perf_verdict": "PASS",
  "overall": "PASS"
}
```

**没有 `rollback_to` 字段**。Gate D FAIL 时，Pilot 只有文字描述（"rollback_to 范围限制：phase-4a / phase-3 / phase-2"），无法机械解析。这违反"产物驱动流转"原则：流转决策必须来自结构化产物，而非文档描述。

### 修复方案

**在 gate-d-review.json schema 中新增 `rollback_to` 字段：**

```json
{
  "gate": "D",
  "agent": "Auditor-QA",
  "attempt": 1,
  "test_verdict": "PASS",
  "coverage_verdict": "PASS",
  "perf_verdict": "N/A",
  "overall": "PASS",
  "rollback_to": null,
  "rollback_reason": null
}
```

FAIL 场景示例：

```json
{
  "gate": "D",
  "agent": "Auditor-QA",
  "attempt": 1,
  "test_verdict": "FAIL",
  "coverage_verdict": "PASS",
  "perf_verdict": "PASS",
  "overall": "FAIL",
  "rollback_to": "phase-4a",
  "rollback_reason": "功能测试存在 3 个失败用例，Tester 需补充边界条件测试"
}
```

**Pilot 机械验证 rollback_to 合法性：**

Gate D 的 `rollback_to` 只允许以下值：`null`（PASS 时）、`"phase-4a"`、`"phase-3"`、`"phase-2"`。若 Auditor-QA 输出超出范围的 rollback_to（如 `"phase-0"`），Pilot 拒绝并降级为 `"phase-2"`（最深允许值），同时记录 `[WARN] Gate D rollback_to 超出允许范围，已降级`。

**影响范围：**
- Gate D 章节（Section 5）：更新产物 JSON 示例（添加 rollback_to 字段）
- Section 7.1 状态机：GATE_D_QA_REVIEW 的 FAIL 分支补充 rollback_to 机械解析说明

## 漏洞 U：new_test_files 在所有 Phase 3 回退路径中的状态处理未统一定义

### 问题

当前 Phase 4a 的说明：

> 若 Phase 4a FAIL 回退到 Phase 3，Pilot 将 new_test_files 列表**保留**在 state.json，Builder 修复后 Phase 3.3 / 3.6 仍排除它们。

但以下场景未定义：

**场景：Phase 4a PASS → Phase 4.2 PASS → Gate D PASS → Phase 5 → ... → Gate C FAIL（第二轮）→ Phase 3 回退**

此时 new_test_files 已在 state.json 中，但文档未说明 Phase 3.3 / 3.6 是否排除它们。若此时不排除，Tester 的新测试（尚未通过 Phase 4a 重验证）会进入回归守卫，可能导致误报失败。

### 修复方案

**统一规则：new_test_files 的排除策略适用于所有 Phase 3 回退路径**

无论是哪个阶段触发的 Phase 3 回退（Gate C、Gate D、Optimizer SLA 违规、Phase 3.1/3.2/3.3），Pilot 均遵循以下规则：

- state.json 中的 `new_test_files` 字段一旦写入，在当前 Pipeline 的所有后续 Phase 3.3 / Phase 3.6 执行中**始终被排除**。
- 唯一清空时机：Pipeline 状态变为 COMPLETED，执行毕业操作时。

**在 Phase 4a 说明中扩展现有规定：**

> **new_test_files 全生命周期规则（扩展）：** Tester 新增的测试文件记录到 `state.json.new_test_files`，在以下任一情况下均从 Phase 3.3 / Phase 3.6 的测试范围中排除：
> - Phase 4a FAIL → Phase 3 回退
> - Gate D FAIL → Phase 3 或 Phase 4a 回退（若再次到达 Phase 3）
> - Gate C FAIL → Phase 3 回退（即使 Phase 4a 之前已通过）
> - Optimizer SLA 违规 → Phase 3 回退
>
> 排除规则持续到 Pipeline COMPLETED，此时毕业操作才将它们写入 `regression-suite-manifest.json` 并清空 `state.json.new_test_files`。

**影响范围：**
- Phase 4a 章节（Section 5）：更新 new_test_files 生命周期说明

## 漏洞 V：Phase 4a 未明确要求生成 coverage 数据，Phase 4a.1 隐式依赖 coverage.lcov

### 问题

Phase 4a（Tester）的产物定义只有：

```
产物：.pipeline/artifacts/test-report.json
```

Phase 4a.1（Test Failure Mapper）的执行内容第 2 步：

> 从覆盖率工具的 `coverage.lcov` 输出（或 Istanbul/nyc JSON 报告）中提取每个失败测试涉及的源文件路径。

**coverage.lcov 的生成是 Phase 4a.1 的必要前提，但 Phase 4a 的产物定义中没有提及它。** 如果 Tester 未启用覆盖率收集（这是完全合理的），Phase 4a.1 的第 2 步失败，所有映射返回 `confidence: UNKNOWN`，触发全体回退，精确回退的设计意图完全失效。

### 修复方案

**Phase 4a 产物列表新增 coverage 数据（必须生成）：**

```
产物：
- .pipeline/artifacts/test-report.json
- .pipeline/artifacts/coverage/coverage.lcov   ← 新增（必须）
- .pipeline/artifacts/coverage/coverage.json   ← 新增（Istanbul JSON 格式，可选但推荐）
```

**config.json 新增 `coverage_tool` 配置：**

```json
"testing": {
  "coverage_tool": "nyc",
  "coverage_format": ["lcov", "json"],
  "coverage_output_dir": ".pipeline/artifacts/coverage/",
  "coverage_required": true
}
```

**Pilot 前置检查（Phase 4a FAIL 时触发 Phase 4a.1 前）：**

Phase 4a.1 启动前，Pilot 验证 `coverage.lcov` 存在且非空。若不存在：
- 不启动 Phase 4a.1
- 直接触发全体回退（降级，等同 PARTIAL_MAPPED）
- 日志记录 `[WARN] coverage.lcov 不存在，跳过 Test Failure Mapper，执行全体回退`

**更新 Phase 4a.1 前置条件说明（新增）：**

> **前置要求：** Phase 4a 必须在覆盖率收集模式下运行（由 config.json 的 `testing.coverage_required: true` 强制）。覆盖率数据生成到 `.pipeline/artifacts/coverage/` 目录，Phase 4a.1 从此目录读取。若 coverage.lcov 不存在，Pilot 跳过 Phase 4a.1 直接全体回退。

**影响范围：**
- Phase 4a 章节（Section 5）：更新产物列表，新增 coverage 数据
- Phase 4a.1 章节（Section 5）：新增前置要求说明
- Section 9 config.json：新增 `testing` 配置块

## 漏洞 W：state.json schema 缺少 phase_5_mode 和 new_test_files 字段

### 问题

Section 7.3 的 state.json 示例中缺少两个关键字段：

1. **`phase_5_mode`**：Phase 5 说 "Documenter 读取 `state.json.phase_5_mode` 决定执行内容"，但 state.json schema 中没有此字段。AUTOSTEP_API_CHANGE_DETECTOR 说"写入 phase_5_mode"，但写入什么字段？写入 state.json 还是直接放在 api-change-report.json？当前设计两者都有（api-change-report.json 里有，state.json 里没有），造成来源混乱。

2. **`new_test_files`**：Phase 4a、Phase 7 毕业机制均引用 `state.json.new_test_files`，但 state.json schema 未定义此字段，崩溃恢复后该字段会丢失。

### 修复方案

**在 Section 7.3 state.json 示例中补充两个字段：**

```json
{
  "pipeline_id": "pipe-20250301-001",
  "current_phase": "phase-7",
  "current_agent": "Monitor",
  "mode": "normal",
  "phase_5_mode": null,
  "new_test_files": [],
  "hotfix": {
    "active": false,
    "scope_confidence": null,
    "hotfix_tasks_file": null
  },
  "attempt_counts": { ... },
  "builder_attempt_counts": { ... },
  "max_attempts": { ... },
  "last_checkpoint": "phase-7",
  "active_conditional_agents": { ... }
}
```

**`phase_5_mode` 字段语义说明（新增注释）：**

```json
"phase_5_mode": null
// 值域：null（未到 Phase 5）| "full" | "changelog_only" | "skip"
// 写入时机：AUTOSTEP_API_CHANGE_DETECTOR 完成后，Pilot 同步写入
// 读取方：Documenter（Phase 5 执行策略）、Phase 5.1 Changelog Consistency Checker（判断是否运行）
```

**`new_test_files` 字段语义说明（新增注释）：**

```json
"new_test_files": []
// 写入时机：Phase 4a（Tester）完成后，Pilot 从 impl-manifest.json.new_test_files 同步
// 清空时机：Pipeline COMPLETED 毕业操作完成后
// 读取方：Phase 3.3 Regression Guard、Phase 3.6 Post-Simplification Verifier（排除这些文件）
```

**明确写入时机（在 AUTOSTEP_API_CHANGE_DETECTOR 说明中新增）：**

> **Pilot 写入 state.json（v6 明确）：** AUTOSTEP_API_CHANGE_DETECTOR 完成后，Pilot 读取 `api-change-report.json.phase_5_mode`，同步写入 `state.json.phase_5_mode`。两个文件均保留此字段（api-change-report.json 作为产物归档，state.json 作为运行时状态）。

**影响范围：**
- Section 7.3 状态持久化：更新 state.json 示例，补充 phase_5_mode 和 new_test_files 字段及说明
- Phase 5 章节（AutoStep API Change Detector 部分）：新增 state.json 写入时机说明
